# Pricing Process: From Inputs to Calculated Outputs

## Assembling Market Data

Before any individual swap is priced, the pricer assembles a snapshot of market conditions for the chosen valuation date. This snapshot — called a `MarketData` object — holds three things: the SOFR discount curve, the Fed Funds projection curve, and the full history of daily EFFR fixings. The two interest rate curves are built from the filtered curve file described in the input section. Each curve is represented internally as a set of zero rates at known tenors (pillars), and rates at any intermediate date are obtained by log-linear interpolation of discount factors on a calendar-day axis. This interpolation method ensures smooth, consistent discounting between pillars. A separate instance of the curve is maintained for SOFR and for Fed Funds — a "dual-curve" setup where SOFR governs discounting (converting future cash flows to present value) and Fed Funds governs projection (estimating what future overnight rates will be). The historical fixings file is loaded alongside the curves and held in memory for use during floating leg accrual calculations.

## Building the Swap's Cashflow Schedule

Once market data is assembled, each trade definition is validated and converted into a structured swap object. The first step is schedule generation: the pricer uses each leg's effective date, maturity date, payment frequency, and roll convention to produce the full list of accrual periods from the start of the swap through its final payment. The `roll_convention` field controls the direction of this schedule — whether periods are laid out working forward from the effective date or backward from the maturity date — and whether end-of-month snapping applies (for example, a monthly swap starting on January 31st would have subsequent period boundaries on February 28th, March 31st, and so on under end-of-month conventions). Each period in the resulting schedule is an `AccrualPeriod` storing both the unadjusted boundary dates and the business-day-adjusted versions; which pair is used for the day-count fraction depends on the leg's `adjust` setting. The fixed leg and floating leg produce independent schedules, so a swap with monthly fixed payments and quarterly floating payments simply has schedules of different lengths on each side.

## Pricing the Fixed Leg

The fixed leg's valuation is straightforward. For each accrual period that has not yet been paid (payment date is after the valuation date), the pricer computes a coupon amount equal to the notional multiplied by the fixed rate and the day-count fraction of the period. The day-count fraction is calculated according to the leg's specified convention — for ACT/360 it is simply the number of calendar days in the period divided by 360; for 30/360 it uses a standardized thirty-day-month approximation; for ACT/ACT-ISDA it accounts for leap years. Each coupon is then discounted back to the valuation date using the SOFR curve's discount factor for the payment date. Summing these discounted coupons across all unpaid periods gives the present value of the fixed leg.

For accrued interest on the fixed leg, the pricer calculates the portion of the current period's coupon that has accumulated from the period's start through the valuation date. The accrual is measured inclusively — both the period start and the valuation date are included in the count — which means on the valuation date itself, the accrued amount slightly exceeds the pro-rata share one might expect from a simple fraction. This convention reconciles to Chatham Financial's calculation methodology.

## Pricing the Floating Leg: OIS Compounding

The floating leg is considerably more complex because interest is not set at a fixed rate — instead, it compounds daily based on the published overnight Fed Funds rate over each accrual period. This is the standard OIS (Overnight Index Swap) compounding mechanism.

For each accrual period, the pricer works through every calendar day from the period's start to its end. For each day, it determines the applicable overnight rate: if the relevant fixing date is before the valuation date, it retrieves the actual published rate from the historical fixings data; if the fixing date is on or after the valuation date, it computes a forward rate implied by the Fed Funds curve (using the ratio of discount factors at adjacent days, converted to a simple ACT/360 rate). Weekends and holidays are not skipped — they carry the preceding business day's rate, weighted by the number of days until the next business day (typically three days for a Friday). If the trade specifies a lockout period, the last N days of each period reuse the rate from day N+1-to-last rather than observing new fixings; this is a contractual feature of some trades.

Once all daily rates are assembled for a period, the compounded rate is calculated by multiplying together the growth factors `(1 + r_i × d_i / 360)` for each day i, where `r_i` is the overnight rate and `d_i` is the number of days that rate applies. Subtracting one and annualizing gives the effective coupon rate for the period: `R_comp = (∏(1 + r_i × d_i / 360) − 1) × 360 / D`, where D is the total calendar days in the period. This compounded rate is multiplied by the notional to produce the floating coupon for the period, which is then discounted using the SOFR curve's discount factor for the payment date.

Accrued interest on the floating leg follows the same daily compounding logic, but only runs through the current period from its start up to and including the valuation date. The pricer calculates the accumulated compounded growth factor using all available historical fixings plus any curve-implied rates needed to complete the count through the valuation date.

## Assembling the Swap Valuation

With both legs priced, the pricer assembles the four core output metrics for each swap.

The **dirty price** (reported as "Total Value / NPV") is the present value of all future cash flows from the portfolio's perspective. Its sign depends on the `pay_fixed` flag: if the entity pays the fixed rate, dirty PV equals the floating leg's PV minus the fixed leg's PV; if the entity receives the fixed rate, the sign is reversed. A positive dirty value means the swap is an asset (the entity is owed money on a net basis); a negative dirty value means it is a liability.

The **accrued interest** is the sum of accrued amounts across both legs, net of their respective signs. On the fixed leg side, accrued represents the fixed coupon that has built up in the current period. On the floating leg side, accrued represents the compounded floating coupon that has built up. The two are combined according to the swap's pay/receive direction.

The **clean price** is simply the dirty price minus the accrued interest. It represents the swap's mark-to-market value stripped of the "interest already earned but not yet paid" component, analogous to the clean price of a bond. The relationship `clean + accrued = dirty` holds exactly at all times.

The **DV01** (dollar value of one basis point) measures how much the swap's value changes if interest rates across the entire curve shift up by one basis point (0.01%). It is computed by full revaluation: the pricer bumps both the SOFR and Fed Funds curves simultaneously by one basis point, reprices the swap under the shifted environment, and takes the difference `DV01 = PV_base − PV_bumped`. A positive DV01 means the position loses value when rates rise — the standard sign convention for a pay-fixed swap (where rising rates benefit the fixed receiver). Historical fixings are held unchanged during the bump; only projected future rates and discount factors move. Matured trades are assigned DV01 of zero.

Two additional computed values are produced for analytical purposes: `pv_fixed` and `pv_floating` are the present values of the fixed and floating legs individually, reported before netting. The `par_rate` is the fixed rate that would make the swap's NPV exactly zero given current market conditions, and `rate_diff_bp` is the difference between the contract's actual fixed rate and this par rate, expressed in basis points — a positive number means the fixed rate exceeds the fair market rate, benefiting the floating-rate payer.

## Key Output Behavior

The pricer produces two primary output files for each valuation run, always placed in a self-contained folder named `output/valdate_<val_date>_rundate_<run_date>/`.

The first and always-present output is the production valuation CSV, `IRS_Valuation_<val_date>-00001.csv`. This file matches the KPMG IRS Valuation feed specification exactly and contains 49 columns across three types of rows: a single header row (row 1) identifying the file, a field-name row (row 2), one data row per priced trade (rows 3 through N+2), and a footer row that sums key numeric columns. The clean price, accrued interest, dirty NPV, and DV01 from the pricer flow directly into columns G through J. Other columns carry pass-through reference fields from the trade definition or are derived by the output writer — for example, whether the trade is cleared through CME is detected from the counterparty name and used to populate five separate columns about settlement type and counterparty classification. The NPV sign drives an asset/liability split into two separate columns (DA and DL) and an asset/liability tag. Two CCID accounting identifiers are built by combining the oracle entity code, the Default RC from the entity lookup file, and a natural account code that depends on whether the NPV is positive (asset, code 192001) or negative (liability, code 392001) for the balance sheet CCID, or always 465012 for the PL/OCI CCID.

The second output, `IRS_Netting_<val_date>-00001.csv`, is produced only when both the netting database and the entity reference file are available. This file aggregates trades by netting group ID and computes gross asset and liability exposures at the group level. For each netting group, the gross DA is the sum of positive NPVs within the group and the gross DL is the sum of absolute negative NPVs. If position netting is allowed for the group (per the netting database), a netting amount equal to the minimum of gross DA and gross DL is computed and subtracted from each side to produce net DA and net DL. The 21-column format of this file matches the KPMG IRS Netting feed specification, with its own header, field-name, data, and footer rows. The netting output does not report individual trade valuations — it consolidates them — making it structurally distinct from the valuation CSV even though both draw from the same underlying priced results.

Additional debug outputs (a portfolio Excel workbook, per-trade detail workbooks, per-trade debug workbooks, and Parquet files) are written only when the `--debug` flag is passed on the command line. These are intended for investigation and reconciliation rather than production delivery. Every run also writes a `manifest_<val_date>.json` file recording the run's unique identifier, the valuation date, the execution timestamp, the git commit hash, SHA-256 hashes of input files, per-trade timings, and any warnings or errors encountered — providing a complete audit trail of the run's inputs and outcomes.
