# Input Files: Structure, Format, and Field Selection

## Market Curve Data

The pricer's primary source of market rates is a daily CSV file named `market_environment_YYYY-MM-DD.csv`, stored in the `data/curves/` folder. This file is a raw export from a market data vendor and is deliberately broad — it contains rates across many asset classes, currencies, and instrument types all interleaved together. The file opens with several non-data header rows (a name row, a date row, and property rows), followed by the body of the data, which is structured as two columns: a ticker label in column A and a numeric rate value in column B.

Because the file contains far more than the pricer needs, a strict filtering step is applied immediately on load. Only rows whose ticker matches the pattern `IR.USD-(SOFR|FEDFUNDS)-ON.ZERORATE-<TENOR>.MID` are retained — everything else (equity volatilities, FX rates, foreign interest rates, swaption surfaces) is silently discarded. This filter selects two families of rates: US SOFR overnight zero rates and US Fed Funds overnight zero rates, each available at a range of tenors spanning from overnight through fifty years. These two sets of filtered rows form the SOFR discount curve and the Fed Funds projection curve respectively. The rate values in column B are interpreted as continuously compounded ACT/360 zero rates.

For teams that prefer to supply curves in a simpler format, two alternative input modes are available. In `--pillar-dates` mode, the SOFR and Fed Funds curves are each provided as a separate no-header CSV file (`sofr_YYYY-MM-DD.csv` and `ff_YYYY-MM-DD.csv`), where column A is an explicit calendar date for each pillar and column B is the zero rate. In `--pillar-dates-df` mode, the same two-file structure is used but column B carries a discount factor directly, bypassing any rate conversion step. All three formats handle one edge case automatically: when the valuation date falls on a non-business month-end (such as a weekend date that is also the last calendar day of its month), no market data is published, so the loader automatically steps back to the previous business day's file. The zero curves are still anchored to the actual valuation date; only the rate pillar values are sourced from the prior close.

## Historical Fixings Data

The floating leg of every swap in this portfolio accrues interest based on the daily Effective Federal Funds Rate (EFFR), the overnight rate published by the New York Fed each business day. The file `data/fixings/fixing_cail_USD-FEDFUNDS-ON.csv` contains the historical record of these daily published rates. It is a simple CSV with three columns — a ticker label, a date, and a rate — with one row per business day. The pricer auto-detects whether the file uses a two-column or three-column layout and handles a variety of date formats flexibly.

Every calendar day within a floating accrual period that falls before the valuation date requires a corresponding entry in this file. If any such historical fixing date is missing, the pricer treats it as a hard error for that trade: the trade is skipped, its error is recorded in the run manifest, and pricing continues for the rest of the portfolio. No interpolation, carry-forward, or substitution is performed in place of a missing fixing. This strictness is intentional — silent fallbacks would produce valuation numbers that look plausible but are wrong, which is a more dangerous outcome than a visible error.

## Trade Definition Files

Each interest rate swap in the portfolio is described by a trade definition stored in `data/trades/`. Trades may be defined individually as YAML files (one file per swap) or collectively as rows in a multi-trade CSV file. Both formats carry the same fields; the CSV format additionally supports comment lines beginning with `#` and skips blank rows.

The trade definition fields fall into two sections. The first section contains pricing inputs — fields the calculation engine uses directly. The required economic terms that must be present on every trade are: `trade_id` (a unique string identifier for the swap), `notional` (the face amount of the swap in dollars), `pay_fixed` (a true/false flag indicating whether the portfolio entity is paying the fixed rate), `fixed_rate` (the agreed fixed coupon as a decimal, where 0.054 means 5.4%), `start_date` (the effective date when accrual begins), `maturity_date` (the termination date), `fixed_frequency` (how often the fixed leg pays, expressed as "1M", "3M", "6M", or "1Y"), and `fixed_daycount` (the day-count convention used to calculate the fixed coupon fraction, such as ACT/360, 30/360, or ACT/ACT-ISDA).

Beyond the required economic terms, each leg of the swap also accepts a set of optional convention fields that mirror Bloomberg's SWPM screen settings. These are specified independently for the fixed leg and the floating leg using prefixed names. For both legs, `bus_day_adj` specifies the business day adjustment rule (such as ModifiedFollowing) used when a scheduled accrual date falls on a weekend or holiday. The `eff_date_adj` and `pay_date_adj` fields similarly control adjustments applied to the effective date and payment dates respectively. The `adjust` field determines whether the day-count fraction for accrual is measured between adjusted or unadjusted period boundaries. The `roll_convention` field controls how the schedule of coupon dates is generated — working forward from the effective date or backward from the maturity date — and whether end-of-month rules apply. The floating leg additionally accepts `lockout_bdays` (the number of days at the end of each period where the final fixing is frozen rather than updated) and `reset_lag_bdays` (a lookback lag shifting each observation date backward). All convention fields default to standard Bloomberg OIS settings when left blank, so a trade with only the eight required fields will price correctly under typical market conventions.

The second section of the trade definition contains production reference fields that pass through directly to the output report and do not affect the pricing calculation itself. These fields are: `quantum_deal_number` (the unique identifier in the firm's Quantum system), `oracle_entity_code` (a four-digit code identifying the Amex legal entity on the trade), `notional_currency` (always USD for this portfolio), `intercompany` (a true/false flag rendered as Yes/No in the output), `counterparty_name_quantum`, `current_counterparty` (the legal name of the external counterparty, where the exact string "CME Clearing House" triggers special cleared-trade treatment), `entity_name_quantum`, `reporting_party`, `counterparty_location`, `deal_date` (the trade date, distinct from the effective start date), `netting_id` (a key into the netting database), and `hedge` (either "LH" for a long hedge or "SC" for short, which drives the Hedged Debt MTM output). Of these pass-through fields, `hedge` and `netting_id` have downstream consequences: `hedge` is required, and an LH designation whose inline debt cannot be priced is treated as a hard per-trade error.

A third section, the hedged-debt block, appears on each trade and is used only when `hedge` is "LH". It describes the bond the swap hedges — `debt_deal_number` (the inline key that replaces the old deal-number map), `debt_fixed_rate` (the bond coupon), `debt_notional` (USD outstanding), `debt_settlement_date` (the bond's issue date, which anchors its coupon schedule and is required for LH), `debt_counterparty`, `debt_frequency`, `debt_daycount`, the `debt_*` mirror of the fixed-leg conventions (defaulting to a bond-appropriate `debt_adjust=pay` and `debt_principal_exchange=end`), and four descriptive output-only fields (`debt_gaap_category`, `debt_instrument`, `debt_rate_type`, `debt_cusip`). The bond's maturity is taken from the trade's own `maturity_date`, and its valuation coupon is `debt_fixed_rate` net of the swap's `floating_spread`.

## Supporting Reference and Lookup Files

Three additional reference files complete the input set, all focused on the production output rather than the pricing engine itself. The `data/entity/Entity_Reference_Report.csv` file maps each Oracle entity code to a Default RC (responsibility center) number; these two values are combined to build the CCID accounting identifiers that appear in the output. Without this file, the CCID columns in the production output will be left blank.

The `data/entity/Netting_Database.csv` file defines the netting groups that aggregate multiple trades for regulatory reporting. Its format is slightly unusual: the first row is a free-form descriptive title, the second row contains the column headers, and trade data begins on the third row. The relevant fields extracted from this file are `netting_id`, `cash_flow_netting_allowed`, `position_netting_allowed`, `netting_entity`, and the Amex legal entity name. Any row in the netting database for an FX instrument is silently skipped. A `netting_id` referenced by a trade that does not appear in this file is treated as a hard input error.

Hedged debt no longer requires separate input files. Previously, an external `Deal_Numbers.csv` map and a `Deal_Summary_YYYY-MM-DD.xlsx` (produced by another team) supplied each debt's clean value and outstanding balance. Now every LH trade carries its bond's terms inline in the hedged-debt block, and the pricer values that bond itself each run — using the same fixed-leg model and SOFR discount curve as the swap — to produce its Clean, Accrued, and Dirty. Those computed values feed the Hedged Debt MTM output (AW = Clean + USD Outstanding) and are also written out as a `Debt_Summary_YYYY-MM-DD.csv` run artifact for review. This keeps the debt marks synchronised with the firm's own model and input data rather than depending on an external feed.
